CS370
For this project, I was given a good amount of starter code that already handled the maze environment and some of the backend pieces, like how the agent interacts with the maze and how experiences are stored. My main job was to build out the Q-training part so the pirate could actually learn how to find the treasure. I wrote the logic that lets the agent decide between exploring new paths or using what it already knows, and I set up the training loop so it could learn from its past actions using deep Q-learning. Basically, I took the pieces that were already there and connected everything so the agent could improve over time and eventually solve the maze consistently.

This project ties into what computer scientists do pretty well. A big part of the field is solving problems that aren’t always straightforward, and this was a good example of that. Instead of just telling the agent exactly where to go, we had to create a system that could figure it out on its own. That kind of thinking is used in a lot of real-world situations, like navigation systems, AI assistants, or anything that needs to learn from data and improve over time. It matters because a lot of modern technology depends on systems that can adapt instead of just following fixed instructions.

When I approach a problem like this, I try to break it down into smaller pieces first. For example, I had to understand how the maze worked, how the agent moves, and how rewards influence learning before I could put it all together. From there, it was a lot of trial and error—testing things, seeing what worked, and fixing what didn’t. That process is pretty typical in computer science. You don’t always get it right the first time, so being able to troubleshoot and adjust is really important.

Even though this was just a game, there are still ethical things to think about. In real-world AI systems, the same kind of logic could be used in more serious situations, so it’s important to make sure things are working correctly and fairly. That includes thinking about things like bias, reliability, and how systems might impact users. There’s also a responsibility to build something that actually works well and doesn’t cause problems for the company or the people using it. So even in a simple project like this, it’s a good reminder that the decisions we make as developers can have bigger impacts.
